# Перенос медиа (бакет `media`) на Cloudflare R2 — финальная архитектура и план внедрения

Проект: Albums (`D:\Albums\`, Supabase ref `rizveurkjpcwrmbtoawj`, прод `https://vgametikok.github.io/albums/`, dev `http://localhost:5085`). Это финальная редакция после трёх ревью (безопасность, полнота, эксплуатация); файлы пока не менялись. Список отклонённых замечаний — в конце документа. Составлено 2026-07-21.

---

## 1. Обзор и потоки: до / после

Главная идея переноса: байты файлов уезжают в приватный R2-бакет `albums-media`, а вся модель доступа (RLS — row-level security, построчные правила доступа в Postgres; предикат `can_view_media`; RPC — хранимые процедуры, отдающие пути) остаётся в Postgres нетронутой. Роль «двери к байтам», которую сейчас играет пара «политика `media_read_p` → `can_view_storage_media`», берёт на себя новая edge-функция `r2-sign`: она проверяет право на просмотр тем же самым предикатом `can_view_media` (через новый серверный RPC) и выдаёт presigned URL (заранее подписанную ссылку, работающую у любого предъявителя ограниченное время) к S3-эндпоинту R2. Бакет `avatars` не трогаем вообще — он публичный, дешёвый и работает.

### Поток «загрузка» (editor.html, posts.html, join.html)

- **До:** `uploadMedia` (upload.js:144-227) жмёт фото/готовит видео → `sb.storage.from('media').upload('${uid}/${id}/orig.ext')` → то же для thumb → insert строки в таблицу `media`. Право на запись гарантирует storage-политика `media_write` (свой uid-префикс).
- **После:** `uploadMedia` после подготовки байтов зовёт `POST /functions/v1/r2-sign/sign-upload` с метаданными (kind, размер, contentType, ext); функция сама проверяет JWT пользователя, квоту (через леджер резервирований, см. §6), тип и размер, сама конструирует ключ `r2/<uid>/<mediaId>/orig.<ext>` (клиент путь не выбирает — этим закрывается запись в чужой префикс) и возвращает два presigned PUT (оригинал + миниатюра) с TTL 15 минут и пришпиленным Content-Type. Клиент делает два обычных `fetch(putUrl, {method:'PUT', body, headers:{'Content-Type': ...}})` прямо в R2 — **сначала миниатюру (она ≤ 2 МБ и заливается за секунды), потом оригинал**: так исчезает сценарий «оригинал лился 14 минут, подпись миниатюры истекла». Затем, как и раньше, insert в таблицу `media`, причём **поле `id` строки обязано браться из `mediaId` ответа функции, а не генерироваться локально** (сейчас клиент сам делает `crypto.randomUUID()` в upload.js:153 — в r2-ветке эта генерация не используется). Это критичный инвариант: `sign-view` извлекает media-id из второго сегмента пути, и при расхождении id строки и id в ключе медиа никогда не подпишется, backfill постера не сойдётся, а свип-скрипт посчитает объект сиротой и удалит его — потеря данных при живой строке. `storage_path` теперь начинается с `r2/`. RLS на insert не меняется. Правило ретрая: **любой неуспех PUT (обрыв сети, таймаут) — это новый вызов `sign-upload`, старый URL повторно не используется** (байты уже в памяти, повторной обработки файла нет).

### Поток «просмотр батчем» (лента, профиль, альбом, календарь, поиск, композер)

- **До:** все 17 вызывающих точек идут через единственный хелпер `signUrls` (ui.js:82-96) → `sb.storage.from('media').createSignedUrls(chunk, 3600)`.
- **После:** `signUrls` остаётся единственной точкой и единственным местом ветвления. Он делит входной массив путей на две корзины по префиксу: пути без `r2/` — как раньше в Supabase Storage; пути с `r2/` — одним батчем в `POST /functions/v1/r2-sign/sign-view`. Ответы сливаются в одну map `path → signedUrl`, и ни один из 17 вызывающих файлов не меняется. Анонимный гость работает: запрос уходит с publishable-ключом проекта в заголовках, функция получает viewer = null и подписывает только публично видимое по `can_view_media`. Важное отличие от первой редакции: **функция деплоится с `verify_jwt=false`** (см. §2) — ключ проекта `sb_publishable_…` (config.js:4) не является JWT, и включённый шлюзовой verify_jwt отдал бы 401 любому гостю до входа в функцию.

### Поток «одиночный просмотр» (lightbox, видео, аудио)

Отдельного потока нет — lightbox и `<video>`/`<audio>` берут URL из той же map, которую построил батч `signUrls` при рендере страницы. Range-запросы (частичная догрузка) видео работают: подпись presigned GET не включает заголовок `range` (он в списке UNSIGNABLE_HEADERS библиотеки aws4fetch), R2 отвечает 206 на одиночные диапазоны, а `<video src>` ходит в no-cors режиме, так что CORS для проигрывания не нужен. Новое против первой редакции: поскольку остаточная валидность подписи 5–15 минут (против нынешних 3600 секунд), перемотка видео или нажатие play в плеере озвучки (`<audio preload='none'>`, narration.js:170) спустя >15 минут после рендера получила бы 403. Закрываем это **дешёвым обработчиком переподписи**: хелпер `attachMediaRefresh(el, path)` в ui.js вешает на `error` медиаэлемента однократный повтор `signUrls([path])` с подменой `src` и восстановлением позиции; подключается в двух местах — видео в lightbox и плеер озвучки (одна строка в narration.js). Ссылку это не удлиняет — просто выдаётся новая по тем же правилам авторизации.

### Поток «просмотр модератором» (moderation.html → mod-api)

- **До:** `action:'open'` → `mod_open_subject` отдаёт сырые пути → `action:'sign'` → `createSignedUrls(paths, 600)` под service-ключом.
- **После:** в `action:'sign'` внутри mod-api добавляется та же вилка по префиксу: `r2/`-пути подписываются aws4fetch-ом (секреты R2 в Supabase общие на весь проект — mod-api их видит через `Deno.env`; про скоуп токена см. §6), остальные — как раньше через `sb.storage`. TTL остаётся 600 с. Модератор под service-ключом видит всё, включая `is_private`, — семантика не меняется.

### Поток «backfill постера видео» (editor.js:339 → upload.js:234-258)

- **До:** одиночный `createSignedUrl(storage_path, 300)` → fetch всего ролика → canvas-кадр → upload thumb с `upsert:true` → update `media.thumb_path`.
- **После:** для `r2/`-строк: GET-ссылку на оригинал даёт `sign-view` (тот же батч-маршрут, один путь), PUT-ссылку на постер даёт `sign-upload` в режиме `target:'thumb'` с `mediaId` существующей строки (функция проверяет, что строка принадлежит вызывающему). Для легаси-строк ветка старого кода сохраняется без изменений — backfill обязан работать для обоих бэкендов, потому что старые видео без постера живут в Supabase.

### Поток «озвучка» (narration.js)

Озвучка заливается через общий `uploadMedia` (narration.js:9, 67) и читается через `signUrls` по пути из RPC `narration_get` — оба конца достаются почти бесплатно. Две оговорки, вскрытые ревью: (1) запись с микрофона даёт blob с типом вида `audio/webm;codecs=opus` (Chrome) или `audio/ogg; codecs=opus` (Firefox), и сегодня из-за отсутствия такого ключа в словаре `EXT` (upload.js:21-25) дорожки уходят как `orig.bin` — перед отправкой в `sign-upload` клиент обязан нормализовать тип (`body.type.split(';')[0].trim().toLowerCase()`) и брать расширение по нормализованному значению, см. §4; (2) в whitelist функции включены `audio/ogg` и `audio/wav` — иначе запись на Firefox и загрузка файлов через инпут `audio/*` (narration.js:36) падали бы с 400. Плюс одна строка подключения `attachMediaRefresh` к плееру (см. выше). Старые дорожки (включая существующие `orig.bin`) продолжают подписываться Supabase-ом нетронутой веткой.

## 2. Edge-функция `r2-sign`

Одна функция, роутинг по хвосту URL (`/functions/v1/r2-sign/<route>`), как в mod-api по `action`. Деплой — только multipart `POST https://api.supabase.com/v1/projects/rizveurkjpcwrmbtoawj/functions/deploy?slug=r2-sign` с PAT из `D:\Albums\.supabase-token` (MCP-коннектор видит другую организацию — не использовать).

**verify_jwt: ВЫКЛЮЧИТЬ (`verify_jwt=false` в metadata деплоя).** Первая редакция ошибочно предполагала легаси-формат anon-ключа (HS256 JWT); фактический ключ проекта — `sb_publishable_vpoMQyLN…` (config.js:4, подтверждено ARCHITECTURE.md:181, 204), это не JWT, и шлюз с verify_jwt=true завернул бы и гостя, и service-ключ нового формата `sb_secret_*`. Ровно по этой причине mod-api уже живёт с `verify_jwt=false` (index.ts:10-11). Защита от мусорного трафика — внутри функции: (1) заголовок `apikey` обязан совпадать с publishable-ключом проекта (значение — в env-секрете функции, сравнение обычное, ключ публичный); (2) настоящая аутентификация — `supabase.auth.getUser(bearer)`: юзер есть → viewer = его uid; нет или ошибка → **fail-closed: viewer = null (только публичное), никогда не throw «в открытое»**; (3) rate-limit (ниже). CORS-заголовки — allowlist origins `https://vgametikok.github.io` и `http://localhost:5085`, как в mod-api (index.ts:20-23), но список заголовков СВОЙ: mod-api разрешает только `content-type, x-mod-token` (index.ts:31), а r2-sign обязан отдавать `Access-Control-Allow-Headers: authorization, apikey, content-type` — иначе preflight (предварительный CORS-запрос браузера) срежет оба маршрута ещё до функции. Правило для dev: только `http://localhost:5085`, открытие по `http://127.0.0.1:5085` не поддерживается (как и у mod-api) — зафиксировано, в списки не добавляем.

**Rate-limit (ограничение частоты).** У mod-api он есть (index.ts:43-46, счётчик по хэшу IP), у r2-sign в первой редакции не было — при фактически публичном `sign-view` это дешёвый DoS (каждый вызов = поход в БД + подписи), а спам `sign-upload` позволял бы штамповать presigned PUT и заливать мусор параллельно до всякого insert. Переносим паттерн: definer-RPC `r2_rate_hit(p_key text, p_limit int, p_window_sec int) returns boolean` поверх новой таблицы `r2_rate` (миграция 019); ключи — `view:<ipHash>` (стартово 120/мин) и `upload:<uid>` (стартово 30/мин); превышение → 429. Дополнительно темп записи ограничивает леджер резервирований (§6).

Клиент AWS-подписи — `import { AwsClient } from 'npm:aws4fetch@1.0.20'` с **обязательными** `service:'s3', region:'auto'` (иначе парсинг R2-хоста ломает подпись); версию зафиксировать в deno.json функции. URL — path-style: `https://${R2_ACCOUNT_ID}.r2.cloudflarestorage.com/${R2_BUCKET}/${key}`.

### Маршрут `POST /sign-view` — батч-подпись чтения (auth ИЛИ гость)

Вход: `{ "paths": ["r2/<uid>/<mid>/orig.webp", ...] }`, до 240 путей (потолок медиатеки композера: posts.js:298, 120 элементов × 2 пути), дедуп на сервере.
Выход: `{ "urls": { "<path>": "<signedUrl>", ... }, "exp": 1784000000 }` — пути, не прошедшие проверку или кривые по формату, просто отсутствуют в map (та же «глотающая» семантика, что у нынешнего `signUrls`, ui.js:90-93 — клиентский код к ней готов).

Проверки по шагам: (1) все пути обязаны матчить regex `^r2/<uuid>/<uuid>/(orig|thumb)\.[a-z0-9]{2,5}$`, иначе путь отбрасывается; (2) из путей извлекаются media-id (второй uuid) и одним запросом под service-ключом зовётся RPC `r2_can_view_media_batch(p_media uuid[], p_viewer uuid)` (§3, миграция 019), возвращающий подмножество id, для которых `can_view_media(id, p_viewer)` истинен — буквально тот же предикат, что стоит в storage-политике `media_read_p`, двойной рубеж сохраняется один в один; (3) прошедшие пути подписываются GET-ом. **Load-bearing инвариант (зафиксировать комментарием в коде функции и в миграции): media-id ВСЕГДА заново извлекается из пути; uid-сегмент [1] в авторизации не участвует и ни с чем не сверяется; никакая строка `media`, заявленная клиентом, под проверку не подставляется.** Именно re-derive из пути делает бессмысленной подмену `storage_path` при клиентском insert. И второй инвариант: **`viewer` выводится ТОЛЬКО из проверенного JWT через `getUser` и НИКОГДА не принимается из тела запроса** — приём viewer из body был бы полным обходом авторизации.

**Детерминированные окна подписи (кэшируемость).** aws4fetch принимает кастомную дату подписи `datetime`. Окно 10 минут: `windowStart = floor(nowEpoch / 600) * 600`, `X-Amz-Date = windowStart` в формате `YYYYMMDDTHHMMSSZ`, `X-Amz-Expires = 900`. Внутри одного окна функция выдаёт байт-в-байт одинаковый URL для одного ключа, поэтому HTTP-кэш браузера работает между перерисовками и между страницами MPA (multi-page application — у нас каждый переход это новый JS-контекст, модульный `urlCache` в ui.js:82 умирает, выигрыш даёт только HTTP-кэш). Фактическая остаточная валидность плавает 5–15 минут — ровно заданный владельцем коридор. **Обязательное дополнение (иначе кэшируемость декларативная): в подпись включается query-параметр `response-cache-control=private, max-age=900`** — R2 поддерживает переопределение ответных заголовков, параметр входит в подпись, детерминизм сохраняется. Без него R2 не шлёт Cache-Control, браузер применяет эвристическую свежесть (~10% возраста объекта по Last-Modified), и для только что залитого фото кэш жил бы ~1 минуту — самый горячий сценарий «загрузил → смотришь альбом» терял бы всю экономию. В ответе `exp = windowStart + 900`; клиентский кэш живёт до `exp - 60`. Дата всегда в прошлом/настоящем — ограничение SigV4 «не из будущего» соблюдено.

### Маршрут `POST /sign-upload` — подпись записи (ТОЛЬКО authenticated)

Вход: `{ "kind", "ext", "contentType", "size", "thumbExt", "thumbType", "thumbSize", "target": "orig"|"thumb", "mediaId": "<uuid>|null" }`. Выход: `{ "mediaId", "path", "putUrl", "thumbPath", "thumbPutUrl", "expiresIn": 900 }`.

Проверки по шагам:

1. `getUser` обязан вернуть юзера, иначе 401 — требование ARCHITECTURE.md:170 «auth.uid() is not null», перенесённое в edge-слой.
2. **Whitelist типов — ТОЧНОЕ равенство нормализованного contentType перечисленному множеству, никаких проверок по префиксу `image/*`** (проверка `startsWith('image/')` пропустила бы `image/svg+xml`; SVG со `<script>` при прямом открытии presigned-ссылки исполняется, а после появления custom domain это стало бы same-site XSS; текущий бакет svg не разрешает — 001_init.sql:740-742, строгий whitelist восстанавливает паритет). Множество собрано из фактического словаря `EXT` клиента (upload.js:21-25) плюс m4v: photo — webp/`image/webp`, jpg/`image/jpeg`, png/`image/png`, gif/`image/gif`; video — mp4/`video/mp4`, webm/`video/webm`, mov/`video/quicktime`, m4v/`video/x-m4v`; audio — weba+webm/`audio/webm`, m4a/`audio/mp4`, mp3/`audio/mpeg`, ogg/`audio/ogg`, wav/`audio/wav`; thumb — только jpg/`image/jpeg`, webp/`image/webp`. `image/svg+xml`, `text/html`, `application/xml` и расширение `bin` отвергаются явно; contentType с параметрами (`;codecs=`) сервер нормализует так же, как клиент. Всё вне списка — 400.
3. Лимиты размера: video ≤ 50 МБ (паритет с лимитом бакета), **photo ≤ 30 МБ** (паритет с действующим `LIMITS.photo = 30*1024*1024`, config.js:11, и ARCHITECTURE.md:153 — первая редакция ошибочно ставила 20 МБ и тем молча резала GIF 20–30 МБ, которые сегодня проходят), audio ≤ 20 МБ, thumb ≤ 2 МБ.
4. **Квота через леджер (журнал резервирований), а не через `media.size_bytes`.** Первая редакция считала квоту по `sum(media.size_bytes)`, но это поле пишет клиент (upload.js:222) без каких-либо ограничений — квота обнулялась бы подстановкой `size_bytes = 1`, плюс два параллельных вызова проходили бы проверку одновременно (классический TOCTOU — time-of-check-to-time-of-use, разрыв между проверкой и использованием). Теперь: definer-RPC `r2_reserve_upload(p_owner, p_media, p_size, p_thumb, p_quota)` под advisory-блокировкой владельца (`pg_advisory_xact_lock(hashtext(p_owner::text))`) атомарно проверяет «сумма по леджеру + p_size + p_thumb ≤ p_quota» и вставляет строку в таблицу `r2_reservations(media_id pk, owner_id, size_bytes, thumb_bytes, created_at)`. Квота считается ТОЛЬКО по леджеру, клиентское `media.size_bytes` в ней не участвует (остаётся информационным полем; в 019 на него добавляется check-constraint `size_bytes >= 0`). Превышение → 413 `{error:'quota_exceeded', used, limit}`. Лимит — секрет `R2_USER_QUOTA_BYTES` (стартово 500 МБ), меняется без редеплоя. Семантика квоты — **только R2** (легаси-байты в Supabase Storage ею не считаются, у Supabase свои лимиты; миниатюры — считаются через `thumb_bytes`); честность заявленного размера — §6.
5. Ключ **всегда строит сервер**: `r2/${user.id}/${crypto.randomUUID()}/orig.${ext}` — клиент физически не может подписать чужой префикс; `ext` подставляется в ключ только после прохождения точного whitelist (никакой интерполяции непроверенных значений). Для `target:'thumb'` клиент присылает `mediaId` (валидация uuid-регэкспом до любого использования), сервер проверяет через service-клиент, что строка `media` существует и `owner_id = user.id`, и подписывает только `r2/<uid>/<mediaId>/thumb.<ext>`.

PUT подписывается с `{ aws: { signQuery: true, allHeaders: true } }` и заголовком Content-Type в Request — без `allHeaders: true` aws4fetch молча выкидывает content-type из подписи (он в UNSIGNABLE_HEADERS), и привязки типа не будет; с ним заголовок попадает в `X-Amz-SignedHeaders`, и R2 вернёт 403 на несовпадение. TTL PUT = 900 с фиксированно. Подпись у S3-семантики проверяется на старте запроса, так что медленная заливка 50 МБ, начатая до истечения, должна доехать — для R2 это документально не зафиксировано, поэтому в смоук шага 3 добавлен дешёвый тест: замедленный PUT, стартующий на 14-й минуте жизни подписи.

### Ошибки

Единый формат `{ "error": "<code>" }` + честные HTTP-коды: 400 (кривой вход/тип), 401 (sign-upload без юзера), 403 (thumb чужой строки), 413 (квота/размер), 429 (rate-limit), 405 (не-POST), 500 с логированием в консоль функции (смотреть через Management API logs). `sign-view` пользовательских ошибок почти не имеет — невидимое молча выпадает из map.

## 3. Различение бэкендов: префикс `r2/` в `storage_path`, без новой колонки

**Решение: префикс пути. Значение в БД — дословно ключ объекта в R2**, то есть `storage_path = 'r2/<uid>/<mediaId>/orig.webp'`; строка из БД одновременно и маркер бэкенда, и ключ, трансформации нет.

Обоснование, привязанное к дисциплине миграций проекта:

1. **Новая колонка `media.storage` бесполезна без правки ядра.** Пути до клиента доносят не селекты, а шестнадцать definer-RPC: `feed_albums`, `feed_posts`, `feed_recommended`, `trending_albums`, `get_album`, `get_post`, `get_profile`, `search_all`, `calendar_albums`, `album_invite_peek`, `narration_get`, `my_media`, `my_shared_albums`, `my_videos_without_poster`, `export_my_data`, `mod_open_subject` (список сверен со всеми `create or replace function` в 001–018 — других функций, возвращающих `storage_path`/`thumb_path`, нет). Чтобы клиент узнал бэкенд из колонки, каждый должен отдавать флаг рядом с каждым путём (в `feed_*` путь — ещё и `coalesce(thumb_path, storage_path)`, флаг пришлось бы вычислять по той же ветке coalesce). Почти все живут в 015_core_rebuild — их переопределение означает правку 015 с полным переприменением плюс ломающее изменение сигнатур ⇒ миграция и фронт одним релизом со снапшотом pg_proc (PLAN.md:49-52). Огромный радиус поражения ради одного бита.
2. **Префикс проезжает через все RPC бесплатно уже сегодня**, включая jsonb-выдачи (`slides.path`, `chapters[].media[].path`), coalesce-ветки и `export_my_data`. Ноль правок ядра.
3. **Коллизии исключены конструктивно:** легаси-пути всегда начинаются с uuid владельца (конвенция 001:227), uuid не бывает строкой `r2`. Если `r2/`-путь случайно попадёт в Supabase-ветку, `can_view_storage_media` распарсит сегмент как uuid, не найдёт строку и вернёт false — фейл безопасный.
4. Новые серверные объекты — все в миграции `019_r2_sign_support.sql`, легально (только новые объекты, ядро не переопределяется, `can_view_media(m_id, viewer)` из 015:60 только вызывается): RPC `r2_can_view_media_batch(p_media uuid[], p_viewer uuid) returns setof uuid` (обёртка `select m from unnest(p_media) m where can_view_media(m, p_viewer)`), RPC `r2_owner_usage(p_owner uuid) returns bigint` (сумма по леджеру), таблица-леджер `r2_reservations` + RPC `r2_reserve_upload(...)`, таблица `r2_rate` + RPC `r2_rate_hit(...)`, check-constraint `media.size_bytes >= 0`. Все функции — security definer, `set search_path = public`. **Гранты: не полагаться на наследование от `revoke ... from public` из 001:773 — он на функции будущих миграций не действует, а default privileges Supabase раздают execute каждой новой функции автоматически.** Поэтому в 019 для каждой функции — явный `revoke execute ... from public, anon, authenticated;`; таблицы — RLS включён без политик (deny-by-default). Без этого `r2_can_view_media_batch` стала бы для любого анонима оракулом «видит ли произвольный viewer вот эти media», а `r2_owner_usage` — утечкой объёма хранилища любого пользователя. Заодно на шаге 2 проверяем `proacl` уже существующих `mod_*` из 018 тем же запросом — не открыты ли они по default privileges. Комментарии в миграции фиксируют оба инварианта из §2: viewer — только из JWT в edge-слое; media-id — только re-derive из пути.

Клиентское ветвление — в одном месте, `signUrls` (ui.js), плюс точки в upload.js:

```js
const isR2 = p => p.startsWith('r2/');
// signUrls: toSign.filter(isR2) → r2-sign/sign-view; остальные → createSignedUrls как раньше
```

Всё остальное работает с map `path → url` и префикса не замечает; regex видео-детекта в `thumbEl` (ui.js:296) смотрит на расширение хвоста, которое у r2-путей идентично.

## 6. Квоты и злоупотребления

**Что ограничивает `sign-upload`:** kind/ext/contentType по строгому whitelist с точным равенством (svg/html/xml/bin — явный отказ); размер по kind (video 50 МБ, photo 30 МБ, audio 20 МБ, thumb 2 МБ); rate-limit 30 подписей/мин на uid; **квота владельца по леджеру `r2_reservations`** — атомарное резервирование заявленных `size + thumbSize` под advisory-блокировкой при каждом `sign-upload`, лимит `R2_USER_QUOTA_BYTES` (env-секрет, стартово 500 МБ, меняется без редеплоя). Семантика квоты: только R2-байты (оригиналы + миниатюры по заявленным размерам); легаси-строки Supabase в неё не входят. Ключ строится сервером только в `r2/<auth-uid>/...`. Content-Type пришпилен подписью (`allHeaders:true`).

**Честность заявленного размера.** Presigned PUT не привязывает длину тела (content-length не подписывается, payload = UNSIGNED-PAYLOAD), поэтому клиент может заявить 1 КБ, а залить 50 МБ. Рубежи: (1) квота списывается с леджера в момент подписи и клиентом не редактируется; (2) лимит 50 МБ на файл + rate-limit 30/мин + TTL PUT 15 минут ограничивают темп злоупотребления сверху; (3) свип (ниже) сверяет фактические размеры объектов из листинга R2 с леджером, правит расхождения и репортит систематических «занижателей»; (4) free tier 10 ГБ — общий стоп-кран.

**Секреты и скоуп токена.** Секреты edge-функций в Supabase — общие на весь проект: R2-креды видит и mod-api, и любая будущая функция. Поэтому R2 API-токен создаётся **строго bucket-scoped: права Object Read & Write только на `albums-media`**, не account-scoped.

**Осиротевшие объекты** (PUT прошёл, insert упал): объект висит в R2 без строки. Это не дыра доступа (без строки `media` его никто не подпишет), а утечка места. Уборка: **свип-скрипт по запросу владельца, примерно раз в месяц или при приближении к 10 ГБ** — Node-скрипт `D:\Albums\tools\r2-sweep.mjs` (aws4fetch работает и в Node): ListObjectsV2 по бакету → сверка вторых uuid ключей со строками `media` (источник истины — prod-БД через PAT + `POST /v1/projects/.../database/query`) → удаление сирот старше 24 часов; заодно чистка строк леджера без объекта/строки media и сверка размеров. **Обязательная дисциплина свипа — бакет без версионирования, это единственная копия данных:** первый запуск всегда dry-run (только список кандидатов), удаление — вторым запуском с явным флагом; предохранитель «не удалять, если кандидатов больше 20% объектов бакета». Cron не строим — объёмы не те. Смежный факт для беклога: клиент и сейчас не удаляет байты из Supabase Storage при удалении строки `media` (в клиенте нет ни одного `.remove()`) — свип это общий ответ на сирот в обоих бэкендах.

## 7. Точки расширения (НЕ строить сейчас)

- **Гостевая загрузка по инвайт-токену без аккаунта.** Дизайн не блокирует: `sign-upload` — единственная дверь записи, в неё добавится альтернативная аутентификация «валидный инвайт-токен события» (проверка через хэш в `album_invites`, как в `album_invite_peek`). Запись через presigned PUT (а не через storage-RLS, требующую authenticated) стратегически правильная: RLS гостей не пустила бы никогда. **Обязательные условия при постройке:** короткий TTL токена, отдельная per-album/per-token квота в леджере, и ключ, жёстко скоупленный альбомом (вида `r2/<albumId>/guest/<uuid>/...`).
- **NSFW-проверка (Workers AI) публичного контента до ленты.** Точка врезки готова: контент попадает в выдачу только через RPC лент, «объект лежит в R2» отделено от «строка видима». Будущий статус модерации на строке — новая колонка + фильтр в выдачах (осознанная правка 015) плюс воркер, читающий свежие публичные объекты по presigned GET. r2-sign менять не нужно.
- **CSAM-хук.** Cloudflare CSAM Scanning Tool работает на уровне зоны через кэш и не видит трафик сырого S3-эндпоинта — заработает при появлении custom domain для раздачи. До custom domain — ручное реагирование через mod-api. Важно: появление custom domain — момент, когда строгий запрет svg/html в whitelist (§2) станет критичным вдвойне (same-site контекст).
- **Статусы модерации** (черновик → проверка → опубликовано/отклонено → апелляция; CSAM — отдельная ветка без апелляции). Чисто БД-машина состояний; r2-sign участвует одним будущим правилом «не-владельцу подписывать только опубликованное».

**Шаг 9. [владелец, по желанию] Учебный откат** — убедиться, что план отката понятен.

**Откат, если R2 сломается:** (1) мгновенно — `MEDIA_BACKEND='supabase'` в upload.js (одна строка): новые загрузки снова в Supabase, легаси и так читается; (2) уже залитые r2-строки на время инцидента без превью — `signUrls` их молча пропустит, в консоли `console.warn`; (3) если R2 умер навсегда — скриптом перелить r2-объекты в Supabase Storage и `update media set storage_path = replace(storage_path,'r2/','')`. Edge-функцию и миграцию 019 откатывать не нужно — в пассиве они безвредны.

## 9. Риски и их закрытие + экономика

| Риск | Как закрыт |
|---|---|
| Утечка presigned URL (bearer) | TTL 5–15 мин (окно 600 + expires 900) — вчетверо короче нынешних 3600 с; URL не пишутся в БД; `referrerpolicy:'no-referrer'` сохраняем |
| Обход квоты клиентским `size_bytes` / гонка заливок | Леджер `r2_reservations`, пишет только сервер атомарно под advisory-блокировкой; свип сверяет фактические размеры |
| Подмена Content-Type | `allHeaders:true` → Content-Type в подписи → 403; whitelist точным равенством |
| SVG/HTML со скриптами как «фото» | Явный запрет svg/html/xml/bin; равенство, не префикс — критично к моменту custom domain |
| Заливка в чужой префикс | Ключ строит только сервер из `auth.uid()` |
| Рассинхрон `media.id` и id в ключе | Правило «id строки = mediaId ответа sign-upload» + смоук шага 5 |
| DoS / спам подписей | Rate-limit 120 sign-view/мин на ipHash, 30 sign-upload/мин на uid; apikey-гейт |
| Потеря постера | Паритет с текущим: `my_videos_without_poster` + backfillPoster доделает |
| Протухание подписи под живым video/audio | `attachMediaRefresh`: onerror → повторный `signUrls` → подмена src |
| Ошибочное удаление свипом | Dry-run + удаление вторым запуском с флагом + порог «>20% бакета — стоп» |
| «Двойной бэкенд навсегда» | Принято осознанно; **письменный триггер второго инкремента (перенос легаси): egress Supabase > 4 ГБ/мес ИЛИ storage > 80% лимита** |
| Переполнение free tier 10 ГБ | Пер-юзерная квота; свип; выход за 10 ГБ не ломает сервис — включается $0.015/ГБ |
| Ошибка в r2-sign кладёт просмотр | Только r2-строки; легаси не зависит; `console.warn` делает инцидент видимым |

**Экономика на тарифах R2** (Standard; egress $0). Допущения: средний активный ~50 МБ хранения, ~1500 просмотров медиа/мес; кэш браузера в расчёт не взят — консервативно.

| Метрика | 1k MAU | 10k MAU | Тариф |
|---|---|---|---|
| Хранение | ~50 ГБ → **$0.60/мес** | ~500 ГБ → **$7.35/мес** | $0.015/ГБ-мес после 10 ГБ |
| Class B (GetObject) | ~1.5 млн → **$0** | ~15 млн → **$1.80/мес** | $0.36/млн после 10 млн |
| Class A (PutObject) | ~0.1 млн → **$0** | ~1 млн → **≈$0** | $4.50/млн после 1 млн |
| **Итого R2** | **≈ $0.60/мес** | **≈ $9–10/мес** | |

**Смежные квоты вне R2:** каждый `sign-view` — инвокация Supabase Edge Function (free tier 500 тыс./мес) + `getUser` + RPC; на 1k MAU (~100–200 тыс./мес) — в бесплатном, на 10k MAU — закладывать в бюджет Supabase-плана. Ориентир Class B: 10 млн / 48 превью ≈ 208 тыс. открытий лент/мес бесплатно даже при нулевом кэше. Легаси-медиа продолжает тратить egress Supabase (free ~5 ГБ/мес) — отсюда триггер второго инкремента.

## Отклонено ревью и почему

1. **HEAD-сверка размера после каждой заливки** — леджер закрывает главную дыру; расхождение ограничено сверху 50 МБ × 30/мин и правится свипом; отдельный finalize — лишняя сложность.
2. **`X-Content-Type-Options: nosniff` через response-переопределение** — у S3-семантики нет такого response-параметра; закрыто строже запретом svg/html/xml на входе.
3. **Убрать `http://localhost:5085` из прод-CORS** — принято как осознанный риск (один проект на dev/прод); вынесено владельцу (§10 п.5).
4. **Добавить `http://127.0.0.1:5085`** — вместо расширения списков зафиксировано правило «dev только через localhost» (паритет с mod-api).
5. **Поднять `X-Amz-Expires` sign-view до 3600** — коридор 5–15 минут это требование приватности; проблема позднего play закрыта переподпиской `attachMediaRefresh`, которая ссылки не удлиняет.
6. **«Запретить bin» как основной путь починки озвучки** — принята нормализация MIME + расширение whitelist (сохраняет текущее поведение); запрет `bin` на сервере оставлен как страховка.
